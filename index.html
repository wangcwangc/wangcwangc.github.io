所用命令解释

nohup命令全名是no hang up，意为不挂起，这在使用远程服务器运行程序的时候非常有用。因为我们在终端中要是把一个程序放在后台，通常在命令尾添加&，但是这样在退出当前用户后，这个程序就会被终止掉，所以要使用nohup。

在标准输入输出流中，0代表标准输入，1代表标准输出，2代表标准错误输出。

>out.txt表示输出到这个文件，而2>&1表示把标准错误输出重定向到标准输出，所以合起来就是，把标准错误输出重定向到标准输出中，然后标准输出到out.txt文件中。

如果不设置输出文件，则会使用缺省值nohup.out。

&为后台运行。

可能用到的其他命令

如果我们想要查看任务状态可以用jobs命令，查看pid信息，需要加参数jobs -l。

但是，如果我们重新登录后，再使用jobs会发现所有的程序都没了，这并不是因为程序终止了，而是jobs清空，可以理解成你不是刚才登录的你了。

所以我们在这种情况下，只能使用ps来查看pid。

其中ps表示显示后台程序aux是ps的三个参数。

  • a表示显示现行终端机下的所有程序，包括其他用户的程序
  • u表示以用户为主的格式来显示程序状况
  • x显示所有程序，不以终端机来区分

grep表示过滤信息，后边python表示只显示每行包含python字符串的那一行。

因为ps命令会显示好几屏的信息，不用过滤的话基本很难找到在哪里。grep根据自己的情况自行设置。

Soot学习之Soot简介（一）

Posted on 2019-11-25
Words count in article: | Reading time ≈

Soot是什么

根据Soot官网的简介，Soot最初是作为Java优化框架开发的。但是后来世界各地的开发者都用Soot来分析、检测、优化和可视化Java以及Android程序。

Soot的输入

目前，Soot可以从以下来源来处理代码：

  • Java源代码
  • Java字节码，包括编译成Java字节码的其他语言，例如Scala
  • Android字节码
  • Jimple中间表示
  • Jasmin，低等级的中间表示（Jasmin同样是该团队的开源项目）

Soot的输出

  • Java字节码
  • Android字节码
  • Jimple
  • Jasmin

根据官方说法，Soot可以从任何输入格式转换为任何输出格式，例如，可以从Android转换成Java或者从Java转换到Jasmin。

Soot提供的分析

  • Call-graph结构（Call-graph construction）
  • 指向分析（Points-to analysis）
  • 定义/使用链（Def/use chains）
  • 模板驱动的程序内部的数据流分析（Template-driven Intra-procedural data-flow analysis）
  • 模板驱动的程序之间的数据流分析，需结合Heros（Template-driven Inter-procedural data-flow analysis, in combination with heros）
  • 结合FlowDroid的污点分析（Taint analysis in combination with FlowDroid），污点就是一些安全风险

Soot的内部中间表示

Soot主要是将程序转换成中间表示（intermediate representation），然后对其进行分析。现在Soot主要提供了四种中间表示用于分析和转换Java字节码：

  • Baf：简化的字节码表示形式
  • Jimple：适合优化的3地址类型的中间表示
  • Shimple：Jimple的SSA变体
  • Grimp：Jimple的聚合版本，适合反编译和代码检查

Jimple是Soot主要使用的中间表示，大多数的分析都是在Jimple上进行的。Soot同样支持添加自定义中间表示。

使用ExpectedException的方式

这种方式的好处就是代码简单，而且可以指定在哪里检测异常，例如若果MyException的构造方法抛出了异常，那么这个测试用例就不会跑通，可以更好的定位异常位置，而且代码相对而言比较清晰明了。

但是这种方式同样存在缺点就是在异常被检测到以后，就会终止这个测试方法，就是说myException.myException();后边的方法就都不会执行了。

使用@Test注解的形式

在注解@Test里边添加expected = IllegalAccessException.class，这样测试方法中的异常就会被捕获，而且这样写的代码最简洁。

但是同样也存问题，就是不能指定断言的信息，而且这样不能定位到方法内部到底是哪里抛出了异常，例如如果MyException的构造方法抛出了异常，这样测试方法同样会跑通。

使用try-catch的方式

这种方式的好处就是可以检测特定位置的异常抛出，并且不影响下文的执行。

但是缺点就是代码太复杂，当然复杂更容易获得完整的测试信息。

以上就是目前Junit 4.x下主流的几种对异常测试的方式，具体使用哪一种可以根据实际情况来定，而@Test(timeout = 4000)中的timeout属性是我自己习惯写上的，并不是此文和前文的规定书写形式，只是为了放在在某些特定情况下测试方法卡住的情况发生。

VO(Value Object 或者 View Object)

值对象：也可以理解为业务对象，存在于业务流程中的对象，用new创建，并由GC回收。

理解举例：如果两个业务之间只想传递两个值，而这两个值存在于两个POJO（Plain Ordinary Java Object），即两个数据表中，直接传两个PO对象就太浪费了，所以可以根据业务需求封装一个VO对象，来进行传递。

POJO(Plain Ordinary Java Object)

POJO：一般只有简单属性，get、set方法，属性和数据库一一对应，基本上数据库有什么属性，持久对象就有什么属性。

PO(Persistent Object)

持久对象：可以用于向数据库中插入、更新、删除信息。
